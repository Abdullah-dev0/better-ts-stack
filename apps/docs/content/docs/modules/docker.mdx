## Overview

The Docker module adds containerization support to your project. It is optional and available for both **Backend API** and **Full-stack** project types.

## What You Get

```
Dockerfile           # Multi-stage build (framework-aware)
docker-compose.yml   # Basic service definition
.dockerignore        # Ignore patterns for the build context
```

<Callout type="info">
  Only these three files are generated. There is no `docker-compose.dev.yml`, no
  database service definition, and no health-check configuration in the
  generated output.
</Callout>

## Generated files

### Dockerfile

The Dockerfile is framework-aware. It generates a Next.js standalone build for full-stack projects and a Node.js multi-stage build for Express backends.

<Tabs items={['Express (backend)', 'Next.js (fullstack)']}>
  <Tab value="Express (backend)">
    ```dockerfile
    # Build stage
    FROM node:20-alpine AS builder
    WORKDIR /app

    COPY package*.json ./
    RUN if [ -f package-lock.json ]; then npm ci; else npm install; fi
    COPY . .
    RUN npm run build || echo "No build script found, skipping build step"

    # Production stage
    FROM node:20-alpine
    WORKDIR /app
    ENV NODE_ENV=production

    RUN addgroup --system --gid 1001 nodejs && \
        adduser --system --uid 1001 expressuser

    COPY package*.json ./
    RUN if [ -f package-lock.json ]; then npm ci --omit=dev; else npm install --omit=dev; fi

    COPY --from=builder --chown=expressuser:nodejs /app/dist ./dist

    USER expressuser
    EXPOSE 3000
    ENV PORT=3000

    CMD ["node", "dist/index.js"]
    ```

  </Tab>
  <Tab value="Next.js (fullstack)">
    ```dockerfile
    # Base stage
    FROM node:20-alpine AS base
    WORKDIR /app
    RUN apk add --no-cache libc6-compat

    # Dependencies stage
    FROM base AS deps
    COPY package*.json ./
    RUN if [ -f package-lock.json ]; then npm ci; else npm install; fi

    # Build stage
    FROM base AS builder
    COPY --from=deps /app/node_modules ./node_modules
    COPY . .
    RUN npm run build

    # Production stage
    FROM node:20-alpine AS runner
    WORKDIR /app
    ENV NODE_ENV=production

    RUN addgroup --system --gid 1001 nodejs && \
        adduser --system --uid 1001 nextjs

    COPY --from=builder /app/public ./public
    COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
    COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

    USER nextjs
    EXPOSE 3000
    ENV PORT=3000 HOSTNAME=0.0.0.0

    CMD ["node", "server.js"]
    ```

  </Tab>
</Tabs>

### docker-compose.yml

```yaml
version: "3.8"

services:
  your-project-name:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
    restart: unless-stopped
```

The service name is set to your project name. You can extend this file manually to add a database service, volume mounts, or environment variable references.

### .dockerignore

Prevents large or sensitive directories from being sent to the build context:

```
node_modules
.env
.env.local
dist
.git
.next
```

## Commands

```bash
npm run docker:build   # docker build -t <projectName> .
npm run docker:up      # docker-compose up -d
npm run docker:down    # docker-compose down
npm run docker:logs    # docker-compose logs -f
```

## Common workflows

### Build and run

```bash
# Build the image
npm run docker:build

# Start in detached mode
npm run docker:up

# View logs
npm run docker:logs

# Stop
npm run docker:down
```

### Extend docker-compose.yml manually

If you need a database container alongside your app, add it by hand after generation. Example for PostgreSQL:

```yaml
services:
  your-project-name:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/myapp
    depends_on:
      - db
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

## Environment variables

The Docker module itself adds no environment variables. Your existing `.env` variables (such as `DATABASE_URL`, `JWT_SECRET`, or `BETTER_AUTH_SECRET`) need to be passed to the container via `environment:` entries in `docker-compose.yml` or a `.env` file used by docker-compose.

<Callout type="warn">
  Never commit your `.env` file to git. Pass secrets via environment variables
  in your deployment platform or use Docker secrets in production.
</Callout>
